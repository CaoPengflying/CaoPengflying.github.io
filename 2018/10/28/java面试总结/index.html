#i/o流

这个结构图是在网上找的一张图。

也就是说在java中的流分为字节流和字符流。
字节流适合读取视频、音频、图片等文件。
字符流适合读取文字内容的文件。
字符流就是按照字符的方式去读取文件和书写文件内容。字节流只能处理8位的字节，不能很好的处理16位的Unicode字符。所以加入了字符流。Reader/Writer，并且还提供了一个类，InputStreamReader和OutputStreamWriter。这两个类可以字符流和字节流中的相互转换。
InputStreamReader，可以将一个字节流中的字节解码成字符。
OutputStreamWriter，可以将字符编码成字节并写入一个字节流。

#集合
list
  list集合常见有arrayList和linkList。这两个集合的底层实现分别是数组和链表。对于这个两个数据结构，数组的查找效率要高一些。两个都是表示一个节点的位置，数组是连续的。而链表是非连续的。所以数组的查找更快。但是删除更慢。删除之后需要将上下的元素移动位置。而链表直接连接下一个节点即可。
set
  list和set的区别。list有序，可重复。set是无序，但是不重复。
map
  子类hashMap、TreeMap、HashTable、以及jdk5以后的concurrent的包
hashMap和HashTable。两者的区别是HashTable使用synchronization修饰的区别。就是在hashTable中使用了synchronized修饰。他是一个线程同步的。这样的效率太低。最终被hashmap所替换掉了。但是目前现状的状况经常有高并发的情况，hashMap又线程不安全，所以在jdk1.5中提供了一个叫ConcurrentHashMap代替了同步的map。
而TreeMap则是可以实现map中的排序，他可以实现key和value的排序。

#线程：如何确保开启的三个线程按照指定的顺序执行
线程中有一个方法是join()。他的作用是调用线程后等待该线程完成后，才继续往下运行
那么实现线程按照指定的顺序执行，就可以在开启线程后，调用join的方法。

当然还有第二个方法，就是并发线程池。在executor中有一个单线程的线程池，而这个线程池的底层原来是先进先出的队列。将三个线程按照顺序放入到单线程的线程池中，就可以确保线程的执行顺序。

该方法对性能的优化可能影响不大，但是他可以找到你项目中执行sql语句丢失了索引或者应该使用索引没有制定索引。

#为什么制定索引能提高查询速度
索引就是在插入数据的时候，事先排好序，从而在查找时可以用二分法查找等高效率的算法。

比如说插入了100万条数据，查找到其中一个，平均需要检索O(n/2)常数不算就是O(n)。而二分法查找的复杂度是（Olog2n）。当n很大的时候能提高效率。
但是在插入数据的时候需要耗费资源，也就是说只有当数据插入的时候要对数据进行提前拍讯。所以当有大量数据的时候用索引。没有大量数据的时候不要用索引。

#sql注入
sql注入指的是，用户在请求中将参数传过来，并且参数中包含了sql语句，以此将sql语句注入到你的项目中，破坏你的项目。
避免sql注入，jdbc中提供了两个类statement和preparedStatment类，第二个类可以采用占位符的方式去设置参数，避免了sql的注入
当然也可以采用过滤参数的方式，在请求到达servlet后，通过获取数据使用正则表达式对参数进行过滤是否含有sql语句，如果没有再将参数传到service层。

#mysql的搜索引擎
1.InnoDB是事务型的数据库的首选引擎，支持事务安全表ACID，支持行锁定和外键，他是Mysql的默认搜索引擎
2.MyISAM搜索引擎:MyISAM基于ISAM存储引擎，并对其进行扩展。适用于在Web、数据仓储和其他应用环境下最常用的存储引擎之一，拥有较高的插入、查询速度，但是不支持事务。
3.Memory存储引擎：将表中的数据存储到内存中，从而提高他的读取速度

#Controller注解和RestController注解的区别
可以说RestController是Controller和ResponseBody的结合。
也就是说RestController需要在方法中返回json数据的时候不需要使用responseBody的注解了。当使用RestController注解的时候视图解析器无法解析jsp和html页面。

#String StringBuilder StringBuffer的区别
字符串的本质是字符数组
String是是使用final修饰过的字符数组。也就是说String是一个不可变的值。如果数据变化，实际上在堆内存中会重新创建一个对象出来，并不会在原有对象本身上修改。这样就会导致产生过多的垃圾，频繁的调用GC导致系统的效率降低。
StringBuilder和StringBuffer两者也是字符串但是他们没有使用final修饰，所以效率会比String类型好一些。
StringBuilder和StringBuffer两者的区别是他们的方法中在StringBuffer中会使用synchronized修饰，同步锁修饰。这样会确保线程安全但是效率会低一些。如果是单线程那么StringBuilder会更好
总结
1String：适合少量字符串操作
2StrinBuilder:适用于单线程在字符串缓冲区进行大量操作
3StringBuffer:适合多线程下在字符缓冲区进行大量操作

#抽象类和接口的区别
jdk1.8前后，两者变化比较大
1.8前：接口中不能有实现的方法，而抽象类可以
1.8后：接口中可以有default和static方法，修饰过的可以有实现体。
抽象类中可以有普通的属性，而接口中只能有常量
抽象类只能是单继承，而接口可以有多实现。并且接口和接口之间是可以多继承的。

#面向对象的7大原则
1.单一职责原则（Single Responsibility Principle）
每一个类应该专注于做一件事情。
2.里氏替换原则（Liskov Substitution Principle）
超类存在的地方，子类是可以替换的。
3.依赖倒置原则（Dependence Inversion Principle）
实现尽量依赖抽象，不依赖具体实现。
4.接口隔离原则（Interface Segregation Principle）
应当为客户端提供尽可能小的单独的接口，而不是提供大的总的接口。
5.迪米特法则（Law Of Demeter）
又叫最少知识原则，一个软件实体应当尽可能少的与其他实体发生相互作用。
6.开闭原则（Open Close Principle）
面向扩展开放，面向修改关闭。
7.组合/聚合复用原则（Composite/Aggregate Reuse Principle CARP）
尽量使用合成/聚合达到复用，尽量少用继承。原则： 一个类中有另一个类的对象。

#Redis缓存
使用Redis做缓存时，如何确保Redis和DB的数据保持一致。
使用Redis做缓存时，流程大概是这样的。请求过来先到Redis根据key来查找，如果key不存在就到db中查找，找到数据后将数据存储到Redis中并将数据返回给用户。确保用户下次查询时到redis中查找。这个时候有个问题就是一旦更新了数据库之后，但是redis中的数据没有同步怎么办？那么在更新了数据库时要同时删除redis中的缓存。
但是可能存在一个极端情况，就是更新了数据库成功但是redis缓存失败，虽然这种情况比较少但是还是要处理，可以将redis中的数据设置失效时间，当数据失效后，就会重新到数据中读取，如果数据没有失效就不用。
